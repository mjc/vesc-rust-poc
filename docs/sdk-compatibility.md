# SDK compatibility matrix

This workspace is an unofficial Rust SDK for controller-resident VESC
packages. The matrix below records the supported proof paths and the limits of
the current STM32 package ABI; it is not a firmware-version promise.

| Surface | Evidence | Boundary |
| --- | --- | --- |
| STM32 function table | `vescpkg-rs-sys` derives 253 slots from the pinned `vesc_c_if.h`; the libclang audit checks field names, offsets, sizes, and callable/scalar shape | The pinned header is the source of truth |
| Older firmware / null tail | `VescIfPresence` preserves holes and scalar words; optional shims return absence or use a documented fallback | Required shims fail closed when a slot is missing |
| Host ABI and mocks | `cargo test -p vescpkg-rs-sys --lib` exercises independent present/absent mock slots and layout contracts | Host pointers are not treated as STM32 pointers |
| ARM/no-alloc sys crate | `cargo check -p vescpkg-rs-sys --target thumbv7em-none-eabihf --no-default-features` | Fixed firmware addresses and inline ARM dispatch remain unsafe internally |
| No-alloc package | `examples/loopback` and `examples/refloat` build without the `alloc` feature | APIs accept caller-owned buffers and slices |
| Allocator-enabled package | `examples/alloc-smoke` installs `VescAllocator` and is covered by the package build gate | Allocation is package-local and firmware-backed |
| Usage-shaped package ports | `examples/loopback` ports the official `examples/extension` `ext-test` contract alongside a typed diagnostic extension, app-data transport, explicit EEPROM read/write through `Firmware::eeprom()`, scoped mutex/semaphore operations, distinct clock-domain reads, leased GPIO, and a bounded SSD1306 page-layout framebuffer matching the official `examples/ssd1306` drawing algorithm; the public `vesc-protocol::buffer` layer also decodes the big-endian/`float32_auto` wire fields used by `custom_data_comm`; Refloat covers a typed custom-config callback backed by the same EEPROM capability | Probe/config addresses are package-owned; persistence is opt-in and remains firmware policy; display transport and pin choice remain board-specific |
| GPIO inventory and leases | `gpio_lease` covers all 13 pinned digital enum values, checked raw-to-pin lookup, exclusive acquire/configure/read/release behavior, all pull/open-drain modes, and package-stop lease invalidation; analog reads and digital writes are lease-only; `stm32_pad` covers explicit raw pad resolution and mode/set/clear forwarding; Refloat's footpad loop holds typed ADC leases | Low-level STM32 pad access remains explicitly unsafe and needs hardware electrical proof |
| CAN status snapshots | `CanStatusStore` owns copied status messages 1-6 for one controller, refreshes independently omittable records, retains each receive timestamp, exposes wrapping age/staleness, and delegates hardware-family ping; focused CAN tests cover all six records | Firmware-owned status pointers never escape; absent/invalid records remain `None` |
| Clock/synchronization | Host tests cover tick rollover, RAII mutex/semaphore cleanup (including panic and timeout paths), typed system-duration waits, checked microsecond sleep chunking, and optional thread-priority absence; loopback includes no-alloc usage-shaped mutex/semaphore calls | Firmware clock domains remain distinct; provider-specific stack proof is separate |
| Controller inputs and safety | `FirmwareInputs` copies remote and PPM snapshots with typed ages, rejects negative/non-finite age or ratio values, exposes output-disable and a fallible timeout snapshot, provides explicit backup persistence, and uses an exclusive shutdown-inhibition guard restored on drop | The pinned ABI exposes backup persistence but no indexed backup-data accessor; malformed timeout/age values are rejected |
| Motor control and telemetry | `MotorOutput` and `MotorTelemetry` use semantic current, ratio, speed, distance, temperature, power, statistics, FOC, and multi-motor selection types; typed gains/scales compose into `MotorCurrent` without primitive extraction; firmware faults map to exhaustive `FirmwareFault::{None, Active, Unknown}` values; open-loop FOC is isolated behind an explicitly unsafe `AdvancedFoc` handle and optional-slot errors | Motor commands remain firmware-authoritative; unknown fault integers fail closed in Refloat; open-loop operations intentionally bypass closed-loop safety and all motor behavior still needs restrained hardware validation |
| Callback ownership | CAN receive callbacks now use typed IDs and callback-scoped payloads; typed PWM, terminal, packet, command-reply, encoder, and IMU registrations use scoped guards or typed callback state; host app-data/custom-config package-global slots use shared claim/release gates; ARM app-data uses the firmware registration result; packet framing supports a borrowed no-alloc registration and an allocator-backed pinned registration for package state; live registrations reject replacement where the ABI reports it; typed trampolines fail closed after lease drop and the package stop hook disables all typed gates before state teardown; package-stop regressions cover command, terminal, encoder, CAN, IMU, PWM, app-data, and custom-config ownership | Raw CAN/PWM callback registration remains an explicitly unsafe escape hatch; ARM custom-config registration has no provider collision status in the pinned ABI; retained provider callbacks are not forcibly replaced; hardware ordering proof remains open |
| IMU and AHRS | `imu` covers readiness, atomic roll/pitch/yaw, finite-validated nine-field calibration snapshots, owned `ATTITUDE_INFO` lifecycle/update snapshots, and exclusive read-callback leases; `ahrs` covers package-owned initial orientation plus normalized Mahony and Madgwick state | Firmware estimator state remains separate from package-owned algorithms; calibration and hardware estimator proof are still open |
| Live firmware settings | `settings` covers source-backed motor current, electrical-speed ramp/brake ceilings, firmware IMU sample rate, mounting rotations, confidence decay, motor pole count, battery chemistry, input/battery-cut voltage, MOSFET/motor temperature, minimum/maximum duty, drivetrain geometry, battery capacity, no-load current, FOC motor constants, and battery-cell setting identifiers; checked reads/writes, semantic `BatteryChemistry`/`BatteryCellCount`/`MotorPoleCount` validation, and explicit `store_cfg` persistence are tested | The typed list is intentionally bounded to source-backed settings; firmware rejection and hardware persistence remain observable boundaries |
| Persistent storage | `eeprom` covers word codecs, offset byte images, partial-word padding, interruption, and write failure; the loopback example now uses a signature-checked fixed image; Refloat loads and persists its 276-byte custom config through the same package EEPROM range; `nvm` covers bounded byte ranges, wipe, capacity, and firmware failures | EEPROM is the package custom range; NVM capability and capacity remain firmware-provided |
| Safe package API | Examples import `vescpkg_rs` only; raw sys calls stay behind crate-private bindings | Open-loop FOC, STM32 pads, variadic `printf`, and raw pointers stay unsafe/internal |
| LispBM values and lists | The pinned LispBM contract exposes byte arrays, explicit string capability through `LispValue::is_string` plus callback-scoped access through `with_str`, and checked proper-list validation/traversal through `is_list`, `list().next_value()`, or its fallible iterator; improper tails return `LispListError::ImproperTail` | The header has no distinct string/list predicates or length-bearing array-data accessor, so strings remain a scoped byte-array borrow and list checks walk the pinned cons ABI |
| VESC Express | Not implemented in this STM32 family | Express needs a separate table, types, loader, and target proof; it must not share STM32 slot order |

## Reproduce the matrix

From the repository root:

```text
nix develop --command cargo check --workspace
nix develop --command cargo test -p vescpkg-rs-sys --lib
nix develop --command cargo check -p vescpkg-rs-sys --target thumbv7em-none-eabihf --no-default-features
nix develop --command cargo nextest run -p vescpkg-rs --features "test-support math"
```

The broader package and artifact gates are exposed by `make check-full` and
`make package`. Hardware-in-the-loop deployment remains a separate, explicitly
requested gate; passing this matrix does not claim device validation.
