# SDK compatibility matrix

This workspace is an unofficial Rust SDK for controller-resident VESC
packages. The matrix below records the supported proof paths and the limits of
the current STM32 package ABI; it is not a firmware-version promise.

| Surface | Evidence | Boundary |
| --- | --- | --- |
| STM32 function table | `vescpkg-rs-sys` derives 253 slots from the pinned `vesc_c_if.h`; every generated `VescIfSlot` retains and tests its exact 1-based header source line and reports its minimum ordered ABI profile; each `VescIfManifestEntry` exposes the originating `c_decl`, minimum `since` profile, bindgen-rendered field signature, STM32 family, function-nullability, and raw safety classification; all 248 callable entries receive generated `raw::resolve_<slot>` shims, with `VescIfAbi::RAW_SHIM_SLOTS` preserving their manifest order and `VescIfAbi::RAW_SHIM_COUNT` checking the inventory size; the libclang audit checks field names, offsets, sizes, and callable/scalar shape | The pinned header is the source of truth; profile, signature, family, nullability, safety metadata, and raw resolver coverage describe the generated table and never replace observed slot presence |
| Older firmware / null tail | `VescIfPresence` preserves holes and scalar words; optional shims return absence or use a documented fallback | Required shims fail closed when a slot is missing |
| STM32 ABI revision profiles | The pinned header marks the 225-slot base table, the 251-slot firmware 6.05 extension, and the complete 253-slot firmware 6.06 extension; `Stm32AbiRevision` derives these profiles from observed slot presence | A profile is an observed table shape, not a firmware-version promise; unknown shapes remain `UnknownCompatible` |
| Host ABI and mocks | `cargo test -p vescpkg-rs-sys --lib` exercises independent present/absent mock slots and layout contracts | Host pointers are not treated as STM32 pointers |
| ARM/no-alloc sys crate | `cargo check -p vescpkg-rs-sys --target thumbv7em-none-eabihf --no-default-features` | Fixed firmware addresses and inline ARM dispatch remain unsafe internally |
| ARM/no-alloc package | `cargo check -p vescpkg-rs --target thumbv7em-none-eabihf --no-default-features` via `make arm-noalloc-check` | The package API remains usable without allocator or math features |
| No-alloc package | `examples/loopback` and the control-loop smoke package build without the `alloc` feature | APIs accept caller-owned buffers and slices |
| Allocator-enabled package | `examples/alloc-smoke` installs `VescAllocator` and is covered by the package build gate | Allocation is package-local and firmware-backed |
| Host alloc + math package | `make alloc-math-test` runs `vescpkg-rs` tests with both `alloc` and `math` enabled | Feature interaction is tested separately from the ARM32 compile matrix |
| Public documentation | `make doc-all` builds workspace rustdoc with all SDK features and no dependencies | Documentation coverage does not claim firmware or hardware behavior |
| Usage-shaped package ports | `examples/control-loop-smoke` demonstrates a no-actuation periodic thread and app-data callback sharing typed state through `with_state_mut`; `cargo-vescpkg control-loop` reuses its shared no-alloc codec, checks setpoint/status/tick/output progress, and reports observed tick-period range/jitter; `examples/loopback` ports the official `examples/extension` `ext-test` contract alongside a typed diagnostic extension, app-data transport, explicit EEPROM read/write through `Firmware::eeprom()`, scoped mutex/semaphore operations, distinct clock-domain reads, leased GPIO, and a bounded SSD1306 page-layout framebuffer matching the official `examples/ssd1306` drawing algorithm; `examples/float-out-boy` is rebuilt by the representative package gate as a full application-shaped SDK consumer; the public `vesc-protocol::buffer` layer also decodes the big-endian/`float32_auto` wire fields used by `custom_data_comm` | Probe/config addresses are package-owned; persistence is opt-in and remains firmware policy; display transport and pin choice remain board-specific; control-loop timing, output evolution, and teardown still need device proof |
| Float Out Boy auxiliary persistence | The source-shaped package main loop refreshes typed motor/config limits before control work, and the 30 Hz auxiliary thread seeds `OdometerMeters`, stores backup data only when not `Running` and strictly beyond 200 m, and retries rejected/absent persistence; focused threshold/failure tests plus ARM32/package gates cover the software path | The recorder trailer, internal LED DMA renderer, and restrained-device persistence proof remain intentionally separate; an absent `store_backup_data` slot is reported by the SDK rather than fabricated |
| Official example coverage boundary | The representative ports cover the official `config`, `extension`, `custom_data_comm`, `thread`, and `ssd1306` usage shapes through safe package APIs; the official `speed_test` benchmark is not a runtime SDK surface, and `ws2812` remains an explicitly low-level STM32 timer/DMA/pad example rather than a safe general-purpose wrapper | `speed_test` timing/printf behavior and `ws2812` peripheral ownership require dedicated benchmark or board-specific unsafe proof; neither is silently advertised as a safe portable capability |
| GPIO inventory and leases | `gpio_lease` covers all 13 pinned digital enum values, checked raw-to-pin lookup, exclusive acquire/configure/read/release behavior, all pull/open-drain modes, and package-stop lease invalidation; analog reads and digital writes are lease-only; `stm32_pad` covers explicit raw pad resolution and mode/set/clear forwarding | Low-level STM32 pad access remains explicitly unsafe and needs hardware electrical proof |
| CAN status snapshots | `CanStatusStore` owns copied status messages 1-6 for one controller, refreshes independently omittable records, retains each receive timestamp, exposes wrapping age/staleness, and delegates hardware-family ping; focused CAN tests cover all six records | Firmware-owned status pointers never escape; absent/invalid records remain `None` |
| Clock/synchronization | Host tests cover tick rollover, RAII mutex/semaphore cleanup (including panic and timeout paths), typed system-duration waits with explicit `SemaphoreWaitOutcome::{Signaled,TimedOut}` results, checked microsecond sleep chunking, and optional thread-priority absence; loopback includes no-alloc usage-shaped mutex/semaphore calls | Firmware clock domains remain distinct; provider-specific stack proof is separate |
| Controller inputs and safety | `FirmwareInputs` copies remote and PPM snapshots with typed ages, rejects negative/non-finite age or ratio values, exposes output-disable and a fallible timeout snapshot, provides explicit backup persistence, and uses an exclusive shutdown-inhibition guard restored on drop; `FirmwareCapabilities::inputs()` and `require_inputs()` are the only public constructors and gate the handle on the complete required remote/timeout slot set; the representative Float Out Boy port consumes this capability directly, while the older fallback-shaped `ControllerInput` wrapper is retained only as deprecated compatibility surface | The pinned ABI exposes backup persistence but no indexed backup-data accessor; malformed timeout/age values are rejected; PPM, backup, output-disable, and shutdown slots remain individually optional |
| Motor control and telemetry | `MotorOutput` and `MotorTelemetry` use semantic current, ratio, speed, distance, temperature, power, statistics, FOC, and multi-motor selection types; battery telemetry includes an owned level/remaining-watt-hours snapshot from the ABI's output-pointer contract; tachometer reads use explicit preserve/reset policies and the typed tachometer setter returns the firmware's previous value; cumulative energy reads use explicit preserve/reset policies, PID-position offset writes use an explicit volatile/persistent policy, release waits return `MotorReleaseOutcome::{Released,TimedOut}`, typed gains/scales compose into `MotorCurrent` without primitive extraction, firmware faults map to exhaustive `FirmwareFault::{None, Active, Unknown}` values, fault descriptions are borrowed for the telemetry-handle lifetime rather than exposed as `'static`, and open-loop FOC is isolated behind an explicitly unsafe `AdvancedFoc` handle with optional-slot errors | Motor commands remain firmware-authoritative; unknown fault integers fail closed; firmware-owned fault strings cannot outlive the telemetry borrow; open-loop operations intentionally bypass closed-loop safety and all motor behavior still needs restrained hardware validation |
| FOC audio | Optional beep, tone, sample playback, and borrowed sample-table leases use checked audio types and fail-closed capability handling; stop operations use an explicit `FocAudioStopMode::{Preserve,Reset}` policy and package shutdown releases retained tables | Firmware audio ownership and output behavior still need restrained non-actuating hardware proof |
| Callback ownership | CAN receive callbacks now use typed IDs and callback-scoped payloads; typed PWM, terminal, packet, command-reply, encoder, and IMU registrations use scoped guards or typed callback state; host app-data/custom-config package-global slots use shared claim/release gates; ARM app-data uses the firmware registration result; packet framing supports a borrowed no-alloc registration and an allocator-backed pinned registration for package state; live registrations reject replacement where the ABI reports it; typed trampolines fail closed after lease drop and the package stop hook disables all typed gates before state teardown; package-stop regressions cover command, terminal, encoder, CAN, IMU, PWM, app-data, and custom-config ownership | Raw CAN/PWM callback registration remains an explicitly unsafe escape hatch; ARM custom-config registration has no provider collision status in the pinned ABI; retained provider callbacks are not forcibly replaced; hardware ordering proof remains open |
| UART lease | `Uart` exposes exclusive ownership, checked byte reads/writes, and explicit `UartDuplexMode::{FullDuplex,HalfDuplex}` selection; the raw half-duplex flag is confined to the FFI shim and package-stop coverage releases the lease | UART wiring and firmware acceptance remain board/device-specific; a missing optional UART slot fails closed |
| IMU and AHRS | `imu` covers readiness, atomic roll/pitch/yaw, finite-validated nine-field calibration snapshots, owned `ATTITUDE_INFO` lifecycle/update snapshots, and exclusive read-callback leases; `ahrs` covers package-owned initial orientation plus normalized Mahony and Madgwick state, with checked constructors/setters returning typed `AhrsParameterError` values | Firmware estimator state remains separate from package-owned algorithms; calibration and hardware estimator proof are still open |
| Live firmware settings | `settings` covers source-backed motor current, electrical-speed ramp/brake ceilings, firmware IMU sample rate, mounting rotations, confidence decay, Mahony/Madgwick estimator gains, g accelerometer offsets, degree/s gyroscope offsets, motor pole count, battery chemistry, CAN mode/baud, AHRS, and shutdown-policy selectors, input/battery-cut voltage, MOSFET/motor temperature, minimum/maximum duty, drivetrain geometry, battery capacity, no-load current, FOC motor constants, and battery-cell setting identifiers; checked reads/writes, semantic `BatteryChemistry`/`BatteryCellCount`/`CanApplicationMode`/`CanBaudRate`/`ImuAhrsMode`/`ImuSampleRate`/`MotorPoleCount`/`ShutdownMode`/IMU gain and offset types, and explicit `store_cfg` persistence are tested | The typed list is intentionally bounded to source-backed settings; firmware rejection and hardware persistence remain observable boundaries |
| Persistent storage | `eeprom` covers word codecs, offset byte images, partial-word padding, and typed `EepromError` results for missing words, address overflow, interrupted reads, and firmware write rejection; the loopback example uses a signature-checked fixed image; `nvm` covers bounded byte ranges, wipe, capacity, and firmware failures | EEPROM is the package custom range; NVM capability and capacity remain firmware-provided |
| Safe package API | `make safe-example-check` verifies the representative examples import no `vescpkg-rs-sys` surface and declare `#![forbid(unsafe_code)]`; raw sys calls stay behind crate-private bindings | Open-loop FOC, STM32 pads, variadic `printf`, and raw pointers stay unsafe/internal |
| LispBM values and lists | The pinned LispBM contract exposes byte arrays, explicit string capability through `LispValue::is_string` plus callback-scoped access through `with_str`, checked proper-list validation/traversal through `is_list`, `list().next_value()`, or its fallible iterator, and typed `LispFlatValueError` results for flat-value append/finish operations; improper tails return `LispListError::ImproperTail` | The header has no distinct string/list predicates or length-bearing array-data accessor, so strings remain a scoped byte-array borrow and list checks walk the pinned cons ABI; flat-value ownership still transfers only through successful `unblock_flat` |
| VESC Express | `vescpkg_rs_sys::express` provides the pinned v1 constants, named 32-bit slot map, independent `ExpressLibInfo`/`express_native_start!` loader entry contract, target metadata/name parsing and XIP-versus-relocatable load-kind contract, target-selected XIP/relocatable image views, fail-closed loader, unsafe raw resolver, checked runtime clocks/get-arg boundary, RAII synchronization, owned allocation handles, typed LispBM/evaluator/registration calls, ownership-scoped flat values whose append/finish rejection is a typed error, and no-alloc image/container validators | The package builder and CI matrix intentionally target ARM32 (`thumbv7em-none-eabihf`) only; Express compiler/linker/package integration is deferred separately, and hardware proof remains open. The Express table/image formats must not share STM32 slot order or loader assumptions |

## Reproduce the matrix

From the repository root:

```text
nix develop --command cargo check --workspace
nix develop --command cargo test -p vescpkg-rs-sys --lib
nix develop --command cargo check -p vescpkg-rs-sys --target thumbv7em-none-eabihf --no-default-features
nix develop --command cargo nextest run -p vescpkg-rs --features "test-support math"
```

The broader package and artifact gates are exposed by `make check-full` and
`make package`. Hardware-in-the-loop deployment remains a separate, explicitly
requested gate; passing this matrix does not claim device validation.
